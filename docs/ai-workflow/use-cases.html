kgx — Use cases

Four real-world patterns: mount vendor truth once, layer internal knowledge on top, and let agents query or extend packs without rebuilding your entire doc corpus every shift.

On-call debugging Incident memory Trust & compliance New team bootstrap

At a glance

1. On-call with vendor + runbooks

3am S3 policy alert — agent searches AWS pack and Confluence overlay, not the whole internet.

2. Post-incident institutional memory

Agent writes a postmortem entity into an overlay; the next engineer finds it with kgx query.

3. Governed agent context

Immutable vendor cores, provenance tiers, and validation before any model sees a pack.

4. Day-one team context

Export Confluence, emit AWS pack, init workload overlay — Cursor mounts all three.

On-call debugging with layered packs

Alex · platform engineer · payment pipeline

PagerDuty fires on S3 AccessDenied for payment-uploads. Alex opens the service repo in Cursor. The agent already has two packs mounted — no live crawl of AWS docs, no Confluence login in the IDE.

aws-core.kgpack eng-runbooks.kgpack

  1. Nightly CI refreshes vendor pack

    task spike (or provider pipeline) rebuilds aws-core.kgpack from official docs — version bumped, Git-tagged.

  2. Weekly Confluence export → overlay

    cme spaces … then task confluence-pack produces eng-runbooks.kgpack with internal_overlay provenance.

  3. Agent queries, does not re-ingest

    MCP kgx_query_pack on each path returns scoped hits — faster and more faithful than pasting random search results.

Outcome

Mean time to diagnose drops because the model reasons over curated, versioned context — vendor facts plus your runbook — instead of stale training data or a one-off RAG scrape.

Incident → durable institutional memory

Morgan · engineering lead · checkout platform

A Sev-2 checkout timeout is resolved. Morgan asks the agent to capture what happened in the team's overlay pack — not a wiki page that drifts, not a Slack thread that disappears.

checkout-ops.kgpack

What gets written

Entity incident:2026-05-checkout-timeout with markdown body (timeline, root cause, action items). Relationships incident → affected_service → checkout-api link the graph for future traversal.

Outcome

Knowledge compounds in Git-reviewable packs. The next engineer — human or agent — inherits the same structured artifact, not a vague "check Confluence."

Governed context for security & compliance

Jordan · security architect · regulated workload

Agents are allowed on production systems — but only with approved knowledge sources. Jordan's team publishes vendor packs as immutable vendor_core artifacts and keeps policy overlays separate.

Immutable vendor cores

Agents cannot call MCP upsert on published vendor packs — only overlays owned by the team.

Provenance in the artifact

provenance.json records source_tier so downstream indexers can weight trust.

Validate in CI

kgx validate in pipeline blocks broken manifests before they reach developer laptops.

Outcome

Security gets auditable boundaries: what the model saw, where it came from, and which packs are allowed to change.

Bootstrapping a new service team

Riley · tech lead · greenfield inventory-api

A new squad spins up inventory-api. Riley assembles three packs in an afternoon — vendor AWS, org platform Confluence space, and a thin workload overlay — then points Cursor MCP at the folder.

aws-core.kgpack platform-docs.kgpack inventory-api.kgpack 
# Riley's afternoon
cme spaces https://wiki.example.com/display/PLATFORM
task confluence-pack EXPORT_DIR=~/exports/platform -o packs/platform-docs.kgpack
task spike   # → packs/aws-core.kgpack
kgx pack init packs/inventory-api.kgpack --name inventory-api --pack-id https://example.com/packs/inventory-api
kgx pack entity add packs/inventory-api.kgpack --id adr:001 --type adr --label "Service boundaries" --body-file ./adr-001.md

Outcome

New hires and coding agents share the same curated context stack on day one — without waiting for a platform team to build a custom RAG pipeline per service.
