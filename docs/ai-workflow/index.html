What kgx is (and is not)

kgx is a package manager for operational knowledge — not a RAG platform, not a monolithic enterprise graph, and not an agent harness. Agents mount packs; other systems can index them separately.

Portable packs

Git-friendly directory bundles: manifest, JSONL graph rows, markdown documents, provenance, capability references.

Composable layers

vendor_core immutable vendor docs; overlay internal Confluence/runbooks on top.

Agent surface

kgx-mcp exposes validate, inspect, query, and pack editing over stdio (Model Context Protocol).

End-to-end pipeline

External tools export or crawl sources; kgx providers normalize into records and emit a .kgpack directory. The agent never needs live Confluence credentials if export happened earlier.

Pack anatomy

Everything an agent needs for structured context lives in one directory. Normative schemas live in the repo under schemas/; see pack-spec.md.

my-workload.kgpack/
├── manifest.json   name, version, kind, pack_id, dependencies
├── entities.jsonl   one JSON object per line (id, type, label, …)
├── relationships.jsonl   graph edges (from_id, to_id, type)
├── documents/
│    └── runbook.md
├── provenance.json   source_tier, URIs
└── capabilities.json   MCP / skill references only (no secrets)

Typical agent workflow

The IDE registers kgx-mcp as an MCP server. The model chooses tools based on the user task — no full-doc re-ingestion each session.

  1. Mount context

    Operator (or CI) places one or more .kgpack paths on the machine the agent uses. Cursor config points MCP at python -m kgx.mcp with cwd set to the repo or pack root.

  2. Validate before trust

    Agent calls kgx_validate_pack to ensure manifest schema and loadability before relying on content.

  3. Scoped retrieval

    kgx_query_pack runs keyword search over entity labels and document paths (semantic / vector search can live in a separate indexer reading the same pack).

  4. Inspect structure

    kgx_inspect_manifest returns dependencies, kind, and version so the agent knows whether it is reading vendor core vs internal overlay.

  5. Capture new knowledge (optional)

    After solving an incident, the agent can call kgx_pack_entity_upsert / kgx_pack_relationship_upsert to append to an overlay pack — institutional memory without mutating immutable vendor cores.

Example Cursor MCP config

{
  "mcpServers": {
    "kgx": {
      "command": "python",
      "args": ["-m", "kgx.mcp"],
      "cwd": "/path/to/your/project-or-packs"
    }
  }
}

MCP tools (today)

Tool Purpose
kgx_validate_pack Schema-check manifest; load entities and relationships
kgx_inspect_manifest Return manifest JSON (name, kind, version, dependencies)
kgx_query_pack Keyword search → JSON array of document/entity hits
kgx_pack_entity_upsert Add or replace an entity; optional markdown body → documents/
kgx_pack_relationship_upsert Add or replace a graph edge

Human operator path

The same packs are edited and inspected from the terminal — useful for bootstrapping, CI, and debugging agent output.

Ingest

task spike · task confluence-pack · provider scripts

Verify

kgx validate · kgx inspect · kgx query

Edit

kgx pack init · kgx pack entity add · manifest set

Growing the hydrated core (planned)

kgx keeps packs portable; a separate workspace can rebuild indexes when packs change.

Repo analyzer → suggested packs; Confluence via confluence-markdown-exporter; metagit-style folder scans — all feeding the same emit → pack → mount loop.
